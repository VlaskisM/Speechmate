## Directory Purposes

**`/e/VisualStudioCodeProjects/Speechmate`:**
- Root project directory
- Contains Docker Compose for services (PostgreSQL)
- Root `run.py` acts as wrapper to execute data_ingress module

**`data_ingress/`:**
- Main application package containing the FastAPI web service
- Houses all source code, migrations, and configuration

**`data_ingress/src/`:**
- Core application source code
- Organized by layer (configs, db, services, web)

**`data_ingress/src/configs/`:**
- Purpose: Centralized configuration management
- Contains: Environment-based settings (database connection strings)
- Key file: `db.py` - Loads DB credentials from `.env` and builds connection URLs

**`data_ingress/src/db/`:**
- Purpose: Data persistence layer
- Contains: Database connection, ORM entities, repository pattern implementations

**`data_ingress/src/db/relational/`:**
- Purpose: SQL/relational database specifics
- Key file: `db.py` - Creates async SQLAlchemy engine and session factory
- Entities: ORM model definitions (e.g., Recording table)
- Repositories: Data access abstraction with abstract base classes and concrete implementations

**`data_ingress/src/services/`:**
- Purpose: Business logic orchestration layer
- Contains: Service classes that coordinate between repositories and web handlers
- Key file: `recording.py` - Handles create/read/delete operations, manages sessions, converts ORM objects to Pydantic schemas

**`data_ingress/src/web/`:**
- Purpose: HTTP API layer (FastAPI)
- Contains: Route handlers, request/response validation, API setup

**`data_ingress/src/web/routes/`:**
- Purpose: API endpoint implementations
- Key file: `recording.py` - REST endpoints for recording CRUD operations

**`data_ingress/src/web/schemas/`:**
- Purpose: Request/response validation and documentation
- Key file: `recording.py` - Pydantic models for RecordingCreate and RecordingResponse

**`data_ingress/migrations/`:**
- Purpose: Database schema versioning (Alembic)
- Key file: `env.py` - Alembic configuration that imports ORM models
- versions/ - Autogenerated migration scripts tracked in git

## Key File Locations

**Entry Points:**
- `/e/VisualStudioCodeProjects/Speechmate/run.py`: Root wrapper - spawns data_ingress/run.py in subprocess
- `/e/VisualStudioCodeProjects/Speechmate/data_ingress/run.py`: Starts Uvicorn server (FastAPI app on 0.0.0.0:8000)
- `/e/VisualStudioCodeProjects/Speechmate/data_ingress/src/web/web.py`: FastAPI app instance with routers

**Configuration:**
- `/e/VisualStudioCodeProjects/Speechmate/.env`: Database credentials (DB_HOST, DB_PORT, DB_USER, DB_PASSWORD, DB_NAME)
- `/e/VisualStudioCodeProjects/Speechmate/data_ingress/src/configs/db.py`: Database settings - parses .env and builds connection URLs

**Core Logic by Layer:**
- **Database**: `/e/VisualStudioCodeProjects/Speechmate/data_ingress/src/db/relational/db.py` - Engine and session setup
- **Entities**: `/e/VisualStudioCodeProjects/Speechmate/data_ingress/src/db/relational/entities/recording.py` - ORM Recording model
- **Repository**: `/e/VisualStudioCodeProjects/Speechmate/data_ingress/src/db/relational/repositories/recording.py` - Data access with abstract interface
- **Service**: `/e/VisualStudioCodeProjects/Speechmate/data_ingress/src/services/recording.py` - Business logic and session management
- **Routes**: `/e/VisualStudioCodeProjects/Speechmate/data_ingress/src/web/routes/recording.py` - HTTP endpoints
- **Schemas**: `/e/VisualStudioCodeProjects/Speechmate/data_ingress/src/web/schemas/recording.py` - Request/response models

**Testing:**
- Not present in current structure - tests would typically go in `data_ingress/tests/` directory

## Naming Conventions

**Files:**
- Snake case: `recording.py`, `db.py`, `health_check.py`
- Entity models: Singular, PascalCase: `Recording`
- Service classes: Singular, PascalCase: `RecordingService`
- Repository classes: Singular + Repository suffix, PascalCase: `RecordingRepository`
- Abstract base classes: Prefix with `Abstract`: `AbstractRecordingRepository`

**Directories:**
- Lowercase with plural or descriptive names: `entities/`, `repositories/`, `routes/`, `schemas/`, `configs/`, `migrations/`
- Feature-grouped: `recording.py` modules group by entity type

**Functions:**
- Snake case: `get_all()`, `create_recording()`, `get_by_user_id()`
- Async functions use `async def` prefix
- Private functions prefixed with underscore: `_to_response()`, `_to_response_list()`

**Variables:**
- Snake case: `badge_id`, `file_url`, `user_id`, `recording_id`
- Constants: UPPER_SNAKE_CASE (not observed in current code)

**Types:**
- Pydantic models: PascalCase: `RecordingCreate`, `RecordingResponse`, `BaseResponse`
- SQLAlchemy entities: PascalCase: `Recording`

## Where to Add New Code

**New Feature (e.g., Users):**
- ORM Entity: `data_ingress/src/db/relational/entities/user.py`
- Repository: `data_ingress/src/db/relational/repositories/user.py` (with abstract base)
- Service: `data_ingress/src/services/user.py`
- Schemas: `data_ingress/src/web/schemas/user.py`
- Routes: `data_ingress/src/web/routes/user.py`
- Register routes in: `data_ingress/src/web/web.py` (add `include_router()` call)

**New Endpoint for Existing Feature (e.g., Recording endpoint):**
- Add handler function in: `data_ingress/src/web/routes/recording.py`
- Add/update schemas in: `data_ingress/src/web/schemas/recording.py`
- Add/update service methods in: `data_ingress/src/services/recording.py`
- Add/update repository methods in: `data_ingress/src/db/relational/repositories/recording.py`

**Utility/Helper Functions:**
- Shared utilities: Create `data_ingress/src/utils/` directory with feature-specific modules
- Response helpers: `data_ingress/src/web/schemas/common.py` (use for BaseResponse patterns)

**New Configuration/Settings:**
- Add to: `data_ingress/src/configs/db.py` (extend DatabaseSettings class) or create new config file
- Environment variables: Add to `.env` and documented in README

## Special Directories

**`data_ingress/migrations/`:**
- Purpose: Database schema version control (Alembic)
- Generated: Yes - `versions/` folder contains auto-generated migration files
- Committed: Yes - Migration files are tracked in git for reproducible schema changes
- Usage: Run `alembic upgrade head` to apply migrations, `alembic revision --autogenerate` to create new ones

**`data_ingress/src/db/relational/`:**
- Purpose: SQLAlchemy ORM setup and data access layer
- Generated: No
- Committed: Yes

**`.planning/codebase/`:**
- Purpose: GSD analysis documentation (this file and others)
- Generated: Yes - Created by analysis tools
- Committed: Yes - Documents are tracked to guide future development
